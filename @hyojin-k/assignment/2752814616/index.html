과제 셀프회고

테스트 코드는 해보고 싶었던 부분이고 잘 해내고 싶어서 욕심을 부려서 hard 단계를 일단 해보자하고 시도해봤는데, 초심자가 함부로 넘보기에는 쉽지 않았네요…. 테스트 코드를 처음 접해보는 건데 이지단계까지는 하나씩 문제를 풀어가는 느낌이라 재미있었습니다. 이지 단계까지만…. 미디엄부터는 어떻게 시작해야할지 감이 안잡히더라고요. 그래서 다른 분들의 pr을 많이 참고했습니다. 그런데 같은 문제를 풀더라도 다들 풀어가는 방식이 다양해서 처음에는 감을 잡기 힘들었는데 제가 가장 이해하기 쉬운 방식을 참고해서 적용해보려고 노력했던 것 같습니다.

욕심을 좀 덜고 단계를 낮춰서 진행했다면 hard 단계에서 추가적으로 헤맸던 시간을 테스트 코드의 완성도를 높이거나 테스트 코드를 깊게 학습하는 시간으로 대체할 수 있었을 것 같아 아쉬운 부분이 있습니다. 그래서 일단 테스트 통과 한바퀴를 돌리고 코드를 개선해야지라고 마음 먹었던 부분에 손도 못댔습니다.ㅠㅠ

기술적 성장

  • 각 요소마다 어떻게 접근해야하는지 쉽지 않으면서도 하나씩 찾아가면서 해결해 나갈때마다 재미있었습니다.
  • 한 개의 테스트코드를 짜더라도 그 안에서 생각보다 많은 절차가 필요하더라고요. ‘등록을 하면 목록이 나타난다’ 라는 항목처럼 단순해 보이지만 ‘초기엔 모든 폼이 빈 값이다 → 값을 입력한다 → 버튼을 누른다 → 에러/성공을 반환한다 → 입력값과 목록의 값이 동일한지 확인한다’ 등의 과정들을 하나하나 테스트 해야하는게 까다롭기도 하면서도 재미있었습니다.
  • 테스트에서 자주 작성이 필요했던 이벤트 목록을 위해 팩토리 함수를 작성해서 사용해보았습니다. 
    // 이벤트 생성 팩토리 함수
export const createEvent = ({
  id,
  date,
  title,
  startTime,
  endTime,
  description,
  location,
  notificationTime,
}: Partial<Event>): Event => {
  return {
    id: id || '1',
    date: date || '2025-08-01',
    title: title || `event ${id}`,
    startTime: startTime || '09:00',
    endTime: endTime || '10:00',
    description: description || 'description',
    location: location || 'location',
    category: 'category',
    repeat: {
      type: 'none',
      interval: 0,
    },
    notificationTime: notificationTime || 0,
  };
};

테스트 코드의 항목을 나누는 기준이 궁금합니다. 어느정도 세세하게 나눠야하는지에 대한 기준이랄까요? 테스트의 양이 많아지더라도 모든 발생 가능한 동작에 대해서 테스트 코드를 짜는게 좋은지 궁금합니다.

나누는 기준은 함수마다 다르다고 생각해요. 어떤 함수는 input과 output의 범위가 명확하고, 경계값에 대해 딱 테스트하면 되기 때문에 이에 대한 고민을 할 필요가 없어지기도 하고..?

어떤 함수는 이게 불명확한 경우도 있고 그러니까요 ㅎㅎ 저는 개인적으로 "최대한 많은 케이스"에 대해 일단 작성해보는걸 권장드려요. 효율적인 방법을 찾기 위해선 일단 많이 해봐야하고 체험을 해야 한다고 생각합니다.

일단 커버리지 100%를 다 채워보는거죠. 그렇게 하다보면 "이런 테스트는 없어도 될 것 같은데..?" 라는 생각이 들 수 있어요.

저의 경우 테스트 코드 항목... 이라기보단 컴포넌트에 대한 테스트는 무의미한 경우가 많다고 생각해요. 컴포넌트의 내부 로직을 훅으로 분리한 다음 훅의 input/output 에 대해 테스트하는거죠.

어차피 컴포넌트는 데이터를 표현하는 수단이니까요 ㅎㅎ

handler와 handlersUtils를 작성하기는 했지만 이 둘에 대한 구분이 정확히 되진 않은 것 같습니다. 실제로 코드에서도 handler는 setupServer에서만 사용되고 테스트 코드를 작성할 때는 handlersUtils만 사용했는데 그렇다면 이 둘을 제대로 활용을 못하고 있는 것 같아서요. 둘의 정확한 차이와 각각 어떻게 쓰이면 좋은지 궁금합니다.

차이가 없다고 느낀다면 그냥 합쳐도 된다고 생각해요 ㅋㅋ 제가 설명해서 납득하는 것 보다 효진님께서 느낀게 중요하니까요.

handlerUtils는 각 테스트 케이스에 필요한 별도의 핸들러를 초기화 및 정의하는 용도 handler는 대부분의 테스트에 필요한 기본적인 핸들러를 정의하는 용도
