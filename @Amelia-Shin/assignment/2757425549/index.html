Q. handlersUtils > 아래 여러가지 use 함수는 어떤 역할을 할까요? 어떻게 사용될 수 있을까요?

각 이벤트 생성/수정/삭제를 위한 mock API를 설정하여, 테스트 시작 전에 필요한 mock 환경을 구성할수있습니다. 이미 짜여져 있는 함수를 가져다 쓰는거였지만 미디움에서 편리하게 사용했습니다.. 테스트 환경에서 실제 API를 호출하기에 너무 많은 리소스를 사용한다고 생각이 들고, 실제 서버 대신 가짜 서버와 데이터를 만들어 사용함으로써 더 빠른 테스트를 진행할 수 있다고 생각합니다.

Q. setupTests.ts > 왜 이 시간을 설정해주는 걸까요?

vi.setSystemTime으로 고정된 시간을 설정하여 날짜/시간 의존 로직이 항상 일관된 결과를 반환하도록 해주기 위해서라고 생각합니다. 장점 : 테스트의 예측 가능성과 재현성을 보장!

act 와 waitFor의 차이점

act는 동기, 비동기로 사용 가능 waitFor은 항상 비동기

act와 waitFor은 언제 사용하는게 좋을까?

// 동기 act - 타이머 즉시 진행
act(() => {
   vi.advanceTImersByTime(1000);
});

// waitFor - 실제로 시간이 흘러야 함
  await waitFor(() => {
    expect(screen.getByText('1초 지남')).toBeInTheDocument();
  }, { timeout: 2000 });

Q1) 데이터를 불러오는 방법 중에 1번 방법은 언제 많이 사용하는지 궁금합니다.

=> 1번 방법(setTimeout(resolve, 0))은 "다음 이벤트 루프"로 넘어가면서 pending 상태의 Promise들이 resolve되기를 기다리는 방식이에요. 주로 useEffect나 비동기 작업이 자동으로 실행되는 경우에 사용합니다.

=> 2번 방법은 명시적으로 함수를 호출하는 거고요. 더 명확하고 의도가 분명하기 때문에 2번 방법이 일반적으로 더 좋아요. 1번 방법은 "뭔가 비동기적으로 실행될 것 같은데 정확히 언제 실행될지 모를 때" 사용하는 일종의 해킹 기법입니다.

Q2) 의미있는 테스트 코드를 작성하기 위한 기준이 어떤것인지 궁금합니다.

=> 제일 중요한건 desc에 쓰는 테스트의 의도, 즉 요구사항에 기반하는 것이 중요합니다. 기획서에 해당하는 요구사항이 있었다면 너무 뻔하다고 할지라도 테스트를 하는게 좋습니다.

=> 이번에 보여준 예시에서는 경계값의 테스트인데 달력으로 입력을 받는 Form을 사용한다면 사실 경계값등은 크게 문제가 되지 않을 수도 있겠지만, 코드로 입력하는 엑셀같은거였다면 반드시 검증을 해야겠지요. 사용자 입장에서 해당 요구사항이 틀릴 수 있는 환경인지 아닌지, 그래서 기획상에서 요구사항에 명세를 담아할 정도라면 테스트를 하는게 좋습니다.

=> 그래서 정리하자면 1) 기획관점에서 요구사항 관점에서 중요한 시나리오인가? 2) 그리고 사용자로 인해서 실제로 발생할 수 있는 상황인가? 3) 이 테스트가 실패하면 사용자에게 문제가 되는가? 혹은 무언가 처리가 필요한가? 이런 기준으로 판단해보세요.
