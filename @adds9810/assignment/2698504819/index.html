과제 셀프회고

**"코드는 한 번 쓰고 여러 번 읽힌다"**는 말의 진정한 의미를 깊이 체감했습니다. 처음에는 단순히 코드를 깔끔하게 만드는 것이 목표였지만, 실제로는 **"미래의 자신과 동료를 위해 읽기 쉬운 코드를 작성하는 것"**이 얼마나 어렵고 중요한 작업인지 알게 되었습니다. 특히 과제 발제에서 강조한 **"좋은 코드란, 다른 개발자가 보았을 때 이해하기 쉬운 코드다"**라는 관점이 이번 리팩토링 과정에서 핵심이었습니다. 제가 리팩토링한 바닐라 자바스크립트 코드가 실제로는 여전히 명확하지 않을 수 있고, 새로 만든 React 코드도 다른 개발자가 이해하기 어려운 구조일 수 있다는 점을 직접 경험했습니다.

과제를 하면서 내가 제일 신경 쓴 부분은 무엇인가요?

  • 점진적 리팩토링과 기능 유지 : 처음에는 src/basic/main.basic.js에서 시작하여 문서의 요구사항과 테스트 코드를 참고해 기존의 더티코드를 분석하고, 단계별로 개선해나갔습니다. 특히 docs/02-dirty-code-analysis.md에서 지적된 문제점들(변수 인접성 원칙 위반, 전역 변수 남용, 중복 코드, 매직 넘버 등)을 하나씩 해결하면서도 기존 기능이 정상 작동하는지 계속 확인했습니다.
  • 점진적 개선의 세분화 : 처음에는 AI가 제안하는 단계별 접근 정도면 괜찮겠다고 생각했지만, 실제로는 충돌로 인해 N번을 처음으로 돌아가야 했습니다. 결국 AI가 제안하는 단계에서도 더 세세하게 계획을 세우고 수정을 진행해야 한다는 것을 깨달았습니다. 점진적 개선이 내가 생각한 것보다 훨씬 더 작고 세세한 단위로 나뉘어야 한다는 것을 깨달았습니다. 한 번에 여러 변수를 정리하려다가 순환참조로 인해 전체가 터져버리는 경험을 여러 번 겪었습니다.
  • "읽기 쉬운 코드" 작성의식 : 과제 발제에서 강조한 "코드는 한 번 쓰고 여러 번 읽힌다"는 원칙을 따라, 미래의 자신과 동료가 이해하기 쉬운 코드를 작성하려고 노력했습니다. 하지만 바닐라 자바스크립트 리팩토링 과정에서 여전히 명확하지 않은 부분들이 많이 남아있고, 새로 만든 React 코드도 완벽하지 않다는 것을 깨달았습니다.
  • 더티코드의 실제 어려움 체감 : 특히 모호한 변수명(p, q, amt, sel 등)과 전역변수(sel, cartDisp 등)로 인해 어디서 사용하는지, 무슨 의미인지 파악하기 어려웠습니다. 이런 변수들을 수정하려고 하면 다른 곳에서 의존하고 있어서 터져버리는 경험을 여러 번 겪었습니다. 이 과정에서 **"코드의 의존성과 영향 범위를 파악하는 것"**이 얼마나 중요한지 깨달았습니다.

과제를 다시 해보면 더 잘 할 수 있었겠다 아쉬운 점이 있다면 무엇인가요?

  • DOM 직접 조작 정리 : 현재 main.basic.js 파일에서 DOM 요소들을 직접 조작하는 부분이 완전히 정리되지 않은 상태로 남아있어서, UI/UX 측면에서 개선의 여지가 많습니다.
  • 모듈 간 의존성 관리 : 전역변수 정리를 하면 순환참조로 인해 다른 기능에서 문제가 발생했고, 하나의 원칙을 충족하면 다른 원칙을 충족하지 못하는 경우가 많았습니다. 의존성 주입 패턴을 더 일찍 도입했으면 좋았을 것 같습니다. 특히 더티코드의 전역변수들이 서로 얽혀있어서 하나를 건드리면 연쇄적으로 다른 부분들이 터져버리는 경험을여러 번 겪었습니다.

리뷰 받고 싶은 내용이나 궁금한 것에 대한 질문 편하게 남겨주세요 :)

  • 의존성 주입 패턴의 최적화 : 현재 createSetupEventListeners 함수에서 7개의 함수를 매개변수로 받고 있는데, 이는 함수 시그니처가 복잡해지고 호출하는 쪽에서도 많은 의존성을 전달해야 합니다. 이런 구조에서 의존성을 더 효율적으로 관리할 수 있는 방법이 있을까요? 예를 들어, 의존성 컨테이너 패턴이나 서비스 로케이터 패턴을 도입하는 것이 이 규모의 애플리케이션에서는 과도한가요?
  • DOM 조작 추상화의 적절한 수준 : domElements.js에서 20개 이상의 안전한 래퍼 함수들을 만들었는데, 이는 DOM 조작의 안전성을 높였지만 동시에 코드 복잡성도 증가시켰습니다. 이런 수준의 추상화가 적절한지, 아니면 더 간단한 방식으로 DOM 조작을 안전하게 할 수 있는 방법이 있는지 조언 부탁드립니다.
  • 비즈니스 로직과 UI 로직의 분리 기준 : 현재 businessLogic.js에서 516줄의 복잡한 계산 로직을 담고 있는데, 이 파일이 너무 크고 복잡해졌습니다. 이런 비즈니스 로직을 더 세분화하여 분리할 수 있는 기준이나 패턴이 있을까요? 예를 들어, 할인 계산, 포인트 계산, 재고 관리 등으로 더 세분화하는 것이 좋을까요?

과제 피드백

안녕하세요 지혜, 수고했습니다! 이번 과제는 리팩토링의 진정한 어려움과 가치를 체감하는 것이 목표였어요. 지혜가 써준 회고는 정말 모범생 답안처럼 과제에서 가져갔으면 하는 것들이 다 적혀있어서 참 좋네요. 특히 "코드는 한 번 쓰고 여러 번 읽힌다"는 말의 진정한 의미를 깊이 체감했습니다." 라는 내용이 너무 와닿았습니다.

리팩토링이 단순히 코드를 깔끔하게 만드는 작업이 아니라 "미래의 자신과 동료를 위한 배려"라는 본질을 정확히 파악해준거 같아요. 특히 더티코드 분석 과정에서 p, q, amt, sel 같은 모호한 변수명들로 인해 겪으신 고생 - 결국 내가 겪어보니까 알 수 있는건 데 - 정말 값진(!) 경험이었을 거예요. 그렇게 안 좋은걸 정말 본인이 느껴봐야 알 수 있는데 코드를 더럽게 만든 보람이 생기는군요.

점진적 개선에 대한 깨달음도 인상적입니다. AI가 제안하는 단계에서도 더 세세하게 나누어야 한다는 것을 직접 경험하신 것은 정말 소중한 인사이트예요. 제가 강조했던 것 중 하나로 리팩토링에서 가장 중요한 것이 바로 "작은 단위로 안전하게 개선하기"인데, AI에게 일을 시키면 그런거 없이 자기가 알고 있는 최선을 만들겠다고 하다가 고장나버리잖아요? 내가 가고자 하는 목표를 가기 위해서 점진적인 계획을 촘촘하게 잘 세울수 있는것이 정말 중요하구나 하는걸 느꼈기를 바랍니다.

Q) 의존성 주입 패턴의 최적화 : 현재 createSetupEventListeners 함수에서 7개의 함수를 매개변수로 받고 있는데, 이는 함수 시그니처가 복잡해지고 호출하는 쪽에서도 많은 의존성을 전달해야 합니다. 이런 구조에서 의존성을 더 효율적으로 관리할 수 있는 방법이 있을까요? 예를 들어, 의존성 컨테이너 패턴이나 서비스 로케이터 패턴을 도입하는 것이 이 규모의 애플리케이션에서는 과도한가요?

=> 함수의 적절한 인자의 개수는 3개 이하이며 그 이상이 되는 경우에는 하나의 객체로 받아주는게 좋습니다. 그렇지 않다면 재활용을 하기에 어렵다는 반증이니까요. 그렇지만 단순히 코드를 분리하고 정리하기 위해서 그렇게 사용하는 경우가 있습니다. 함수가 재사용이 아니라 단순히 경계만 만들어주기 위한 도구로 말이죠. 그건 그거 나름대로 괜찮다고 생각합니다.

Q) DOM 조작 추상화의 적절한 수준 : domElements.js에서 20개 이상의 안전한 래퍼 함수들을 만들었는데, 이는 DOM 조작의 안전성을 높였지만 동시에 코드 복잡성도 증가시켰습니다. 이런 수준의 추상화가 적절한지, 아니면 더 간단한 방식으로 DOM 조작을 안전하게 할 수 있는 방법이 있는지 조언 부탁드립니다.

=> jQuery의 패러다임이 DOM 조작의 래퍼함수였죠. 그리고 그게 현대 웹 프레임워크에서는 DOM API를 아예 쓰지 않는 방향으로 진화했습니다. 현대의 관점에서 볼 때에는 낡은 패러다임의 전략이긴 하죠. 그리고 현대의 DOM API는 나쁘지 않기에 래퍼함수 없이 최소한의 DOM API를 직접 사용하거나 완전히 현대적 프레임워크 방식으로 넘어가는게 더 좋은 전략이라고 생각합니다.

Q) 비즈니스 로직과 UI 로직의 분리 기준 : 현재 businessLogic.js에서 516줄의 복잡한 계산 로직을 담고 있는데, 이 파일이 너무 크고 복잡해졌습니다. 이런 비즈니스 로직을 더 세분화하여 분리할 수 있는 기준이나 패턴이 있을까요? 예를 들어, 할인 계산, 포인트 계산, 재고 관리 등으로 더 세분화하는 것이 좋을까요?

=> 비즈니스 로직의 분리 기준은 언제나 요구사항입니다. 요구사항으로 분리를 하다보면 자연스럽게 상위 개념의 그룹이 만들어지고 그러니까 할인, 포인트, 재고 등으로 그룹핑을 할 수 있는거죠. 처음부터 내려 오듯이 세분화를 하면 더 복잡해지기 마련입니다. 하나의 온전한 요구사항을 해결할 수 있는 단위를 분리하고 이들이 함께 있으면 좋을 곳으로 배치한다가 결합도를 낮추고 응집도를 높여가는 좋은 전략입니다.
